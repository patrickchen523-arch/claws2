# game-analyse - 游戏机制全流程分析

输入一个游戏名，自动串联多个 skill 完成从玩法拓扑到机制推理的完整分析。

**产出**：
1. 玩法拓扑报告（gameplay-analyse）
2. 机制扫描报告（mechanism-analyse 模式一）
3. 创新评价卡片（mechanism-evaluate）
4. 机制推理报告（mechanism-reason）
5. 知识卡片 + Game Card + Mechanism Cards + Gameplay Cards（Phase 5 入库产出）

可选：对特定机制做深度分析（mechanism-analyse 模式二）

---

## 一、使用方式

### 触发词
- "分析 [游戏名]"
- "拆解 [游戏名]"
- "看看 [游戏名]"
- "[游戏名] 怎么样"
- "研究一下 [游戏名]"
- "帮我分析 [游戏名]"

### 输入
- 游戏名称（必须）
- 项目背景（可选）——如"我在做一个搜打撤项目"，会影响 evaluate 的判断权重

---

## 二、执行流程

### Phase 0+1：共享信息收集 → 玩法拓扑 → 机制扫描

**做什么**：
合并 Phase 0（玩法拓扑）和 Phase 1（机制扫描）的信息收集为一次搜索，然后顺序执行两个分析步骤。

#### Step 0: 共享信息收集（并行 Sub-agent 搜索）

**目标**：用多个 Sub-agent 并行搜索不同关键词组，汇总结果供后续 Phase 使用。

**关键词分组**（每组分配一个 Sub-agent）：

| 组 | 关键词模板 | 负责 Phase |
|---|-----------|-----------|
| G1 | "[游戏名] 评测 review 玩法 模式 all modes" | Phase 0 + Phase 1 |
| G2 | "[游戏名] 机制 系统 gameplay mechanics systems" | Phase 1 |
| G3 | "[游戏名] wiki 系统详解 guide" | Phase 1 |
| G4 | "[游戏名] 开发者访谈 GDC design philosophy" | Phase 1 |
| G5 | "[游戏名] 全部模式 活动列表 game modes playlist" | Phase 0 |
| G6（可选） | "[游戏名] 攻略 深度评测 核心机制" | Phase 1 补充 |
| G7（可选） | "[游戏名] 社交 匹配 玩家互动 community" | Phase 1 补充 |

**执行方式**：

1. **收集关键词**：根据游戏特性选择 5-7 个适用的关键词组（G1-G5 为必选，G6-G7 按需追加）

2. **并行 Spawn Sub-agents**：每个关键词组分配一个 Sub-agent，指令如下：
   ```
   搜索：<关键词>
   请用 web_search 和 web_fetch 抓取相关内容，返回：
   1. 游戏整体概述（1-2段）
   2. 核心玩法/模式列表
   3. 关键系统/机制描述
   4. 设计亮点或独特机制
   5. 参考链接列表（3-5个高质量来源）
   ```

3. **汇总搜索结果**：主 agent 等待所有 Sub-agent 完成后，将结果合并为一个统一的搜索结果文档，包含：
   - 所有关键词组的搜索结果
   - 高质量来源的网页正文内容摘要
   - 去重后的关键信息

4. **缓存供后续使用**：汇总结果存入上下文，供 Phase 0（玩法拓扑）和 Phase 1（机制扫描）使用，无需重复搜索

**并行搜索模板（复制给每个 Sub-agent）**：

```
# 搜索任务

## 游戏名
[填入游戏名]

## 你的搜索关键词
[填入对应组的关键词]

## 任务
用 web_search 搜索以上关键词，然后用 web_fetch 抓取 2-3 个高质量页面的正文内容。

返回结构化的搜索摘要：
- 游戏整体概述
- 核心玩法/模式
- 关键系统/机制
- 设计亮点
- 高质量参考链接

## 注意
- 优先使用官方Wiki、知名游戏媒体作为来源
- 中英双语搜索（中文关键词 + 英文关键词）
- 不要发明信息，只报告搜索到的内容
```

#### Step 1: Phase 0 — 玩法拓扑分析

**调用**：gameplay-analyse（使用 Step 0 搜索结果，不再独立搜索）

**做什么**：
- 用三条判定线（独立性/操作性/产出性）识别主玩法和副玩法
- 定义主玩法基线（核心循环/目标/约束/节奏）
- 对每个副玩法判断关系类型（变体 vs 独立）并写对应分析

**产出**：`[游戏名] 玩法拓扑报告`

**落盘**：`D:\mechanism-lib\reports\[GameEnglishName]\00-玩法拓扑报告.md`

**完成标志**：报告包含玩法判定表 + 主玩法基线 + 所有副玩法分析

#### Step 2: Phase 1 — 机制扫描

**调用**：mechanism-analyse 模式一（使用 Step 0 搜索结果 + Step 1 的玩法列表，不再独立搜索）

**做什么**：
- 确定游戏品类，加载通用清单 + 品类修正补丁
- 应用品类 SKIP 预设 → 异常检测（详见 mechanism-analyse Step 2.5 和 Step 3.5）
- 按 9 系统分组并发扫描 62 个子模块
- 标注机制出现在哪些玩法中（基于 Step 1 的玩法列表）
- 输出综合叙述（机制特色 / Top 5 / 品类对比 / 风险提示）

**产出**：`[游戏名] 机制扫描报告`

**落盘**：`D:\mechanism-lib\reports\[GameEnglishName]\01-机制扫描报告.md`

**完成标志**：扫描报告包含完整的机制清单表格和综合叙述

**交接给下一步的数据**：整份扫描报告 + 玩法列表

---

### Phase 1.5：知识库检索

**触发条件**：Phase 1 完成后自动执行

**做什么**：分两步检索——先跨品类找相关机制，再读品类先验。

#### Step 1.5a：全局索引检索（跨品类）

从 Phase 1 扫描报告中提取信号，在全局索引中查找跨品类的相关机制卡：

1. **读取全局索引**：`knowledge/_index.md`
2. **从扫描报告中提取查询信号**：
   - 该游戏在解决什么玩家痛点？ → 查 §2.1 痛点索引
   - 想让玩家感受到什么？ → 查 §2.2 体验索引
   - 核心机制模式是什么？ → 查 §2.3 机制模式索引
3. **收集候选机制 ID**：从命中的索引条目中收集 ID，取交集或并集（稀缺品类用并集）
4. **深读候选卡**：在注册表(§1)中查文件路径，深读命中最多的 **3-5 张机制卡**全文
5. **输出跨品类参考**：简要提示给后续 Phase："跨品类参考：[机制名](游戏名) — [一句话相关性]"

#### Step 1.5b：品类先验检索（同品类）

1. 从扫描报告的"基础信息→品类"中提取品类关键词
2. 根据品类映射到对应的知识库子文件夹（如"农场经营" → `knowledge/farming-sim/`）
3. **读取品类总结**：读取 `knowledge/[品类文件夹]/_summary.md`，获取品类共性特征、核心痛点和已有创新模式
4. **按需深入**：如果品类总结中的某个创新模式与当前游戏高度相关，再读取对应游戏的知识卡片

**品类→文件夹映射**：

| 品类关键词 | 知识库文件夹 |
|-----------|------------|
| 农场经营/种田 | farming-sim |
| 生活模拟 | life-sim |
| 城建/城市建造 | city-builder |
| 自动化/工厂 | automation |
| 生存/建造 | survival |
| 三消 | match3 |
| 主题经营/模拟经营 | tycoon |
| 搜打撤/撤离射击 | extraction-shooter |
| 未匹配 | 扫描所有品类文件夹的 _summary.md 寻找最接近的 |

#### 合并输出

将两步的结果合并，简要提示给后续 Phase：
- "同品类已分析过 [N] 款游戏，共性痛点包括 [列表]，已有创新模式包括 [列表]"
- "跨品类参考 [N] 个相关机制卡：[机制名](游戏名) — [一句话相关性]"

**如果知识库为空或无对应品类**：仅执行 Step 1.5a（全局索引仍可跨品类检索），继续 Phase 2

**价值**：
- 让 Phase 2（评价）在判断"品类标配 vs 结构创新"时有更多参考——品类总结中的"创新模式库"提供了该品类已有的创新基线
- 让 Phase 3（推理）在分析协同关系时可以对比"这个游戏的体验支柱和同品类其他游戏有什么不同"
- **新增**：跨品类参考让 agent 看到其他品类的类似机制设计，避免品类壁垒导致的信息盲区

### Phase 2：创新评价

**调用**：mechanism-evaluate

**输入**：Phase 1 产出的扫描报告

**做什么**：
- 读取扫描报告，做完整度检查
- 识别品类原型（对照品类原型词典）
- 三层拆解（原型层 / 创新层 / 包装层）
- 对候选结构创新逐条过 3 条硬性门槛
- 做品类标配排除
- 给出创新定性（换皮无创新 / 原型+结构创新 / 品类突破 / 伪创新）

**产出**：`[游戏名] 创新评价卡片`

**落盘**：将完整评价卡片写入 `D:\mechanism-lib\reports\[GameEnglishName]\02-创新评价卡片.md`

**完成标志**：评价卡片包含品类原型判断、结构创新判据表格、创新定性

**交接给下一步的数据**：评价卡片 + 扫描报告（reason 需要两份）

---

### Phase 3：机制推理

**调用**：mechanism-reason

**输入**：Phase 1 的扫描报告 + Phase 2 的评价卡片

**做什么**：
- 从报告中筛选重点机制（HIGH 优先级 + 结构创新 + ~ 标记的偏离项）
- 层次1：对每个重点机制执行 5 步推理链（设计意图 / 作用机制 / 核心循环影响 / 服务体验 / 代价取舍）
- 层次2：识别体验支柱，绘制机制协同图（乘法/保险丝/弥补/链式关系），识别承重墙与装饰

**产出**：`[游戏名] 机制推理报告`

**落盘**：将完整推理报告写入 `D:\mechanism-lib\reports\[GameEnglishName]\03-机制推理报告.md`

**完成标志**：推理报告包含层次1表格 + 层次2体验支柱和协同关系 + 设计精华总结

### Phase 4（可选）：单机制深挖

**触发条件**：用户在看完前三步产出后，指定某个机制要求深入分析

**调用**：mechanism-analyse 模式二（路径A：从扫描报告取数据）

**做什么**：
- 从扫描报告中提取目标机制的做法概述作为基础
- 补充搜索设计意图和玩家反馈
- 执行 6 步推理链（机制本身 / 直接效果 / 服务目标 / 核心循环影响 / 用户影响 / 代价取舍）

**产出**：`[游戏名] - [机制名] 深析报告`

**落盘**：将完整深析报告写入 `D:\mechanism-lib\reports\[GameEnglishName]\04-[机制名]-深析报告.md`

**注意**：此步骤不自动触发，仅在用户明确要求时执行。

### Phase 5：标准化入库

**触发条件**：Phase 3 完成后自动执行

**做什么**：
从三份报告中提取结构化数据，生成三类入库文件，并更新全局索引。

**引用文件**：
- 入库模板定义：`~/.codemaker/skills/mechanism-analyse/references/entry-templates.md`
- 标签标准化参考：`~/.codemaker/skills/mechanism-analyse/references/标签标准化参考.md`
- 知识卡片模板：`~/.codemaker/skills/mechanism-analyse/references/knowledge/_template.md`
- **全局机制索引**：`~/.codemaker/skills/mechanism-analyse/references/knowledge/_index.md`

#### Step 5a：生成知识卡片（knowledge）

从三份报告中提炼高价值机制精华，写入知识库。

1. **筛选入库内容**：
   - 从 Phase 2（评价卡片）提取：所有被标为结构创新的机制（通过了 3 条门槛的）
   - 从 Phase 3（推理报告）提取：体验支柱、协同关系、承重墙机制、设计精华
   - 品类标配不入库、包装层变化不入库、信息不足的判断不入库

2. **确定品类文件夹**：
   - 根据 Phase 2 的品类原型判断，映射到对应的知识库子文件夹（参见 Phase 1.5 的品类→文件夹映射表）
   - 如果对应品类文件夹不存在，创建新文件夹 + 空的 `_summary.md`

3. **生成知识卡片**：
   - 按 `~/.codemaker/skills/mechanism-analyse/references/knowledge/_template.md` 的格式生成
   - 文件路径：`~/.codemaker/skills/mechanism-analyse/references/knowledge/[品类文件夹]/[游戏名英文小写].md`

4. **如果该游戏已有知识卡片**：
   - 读取已有卡片，合并新发现（不覆盖已有内容，追加新的结构创新）
   - 更新分析日期

5. **更新品类知识总结**：
   - 读取该品类的 `_summary.md`
   - 将新游戏的结构创新追加到"创新模式库"表格
   - 根据新增数据更新"品类共性特征"、"品类核心痛点"、"跨游戏设计启示"
   - 更新"已分析游戏数"和"最后更新"日期

**产出**：`knowledge/[品类文件夹]/[游戏名].md` 知识卡片 + 更新后的 `_summary.md`

#### Step 5b：生成 Game Card（游戏卡）

从三份报告中提取游戏级总览数据，生成一张游戏卡。

1. **读取入库模板**：打开 `references/entry-templates.md`，找到"一、Game Card"章节的模板
2. **按字段来源对照表填充**：每个字段都有明确的 Phase 来源，不需要额外搜索
3. **生成 X-Statement**：综合 Phase 1 基础信息（品类/人数）+ Phase 2（品类原型+结构创新）+ Phase 3（设计精华），按格式生成一句话定位：
   - 格式：`[游玩人数] + [玩法规则(PvE/PvP等)] + [品类定位] + [核心差异点/卖点]`
   - 基础信息需包含：游玩人数、玩法规则(PvE/PvP等)、核心差异点/卖点
   - 示例：`1-8人合作PvE农场经营模拟RPG，通过社区中心跨系统收集束+加工增值链经济，将传统种田从'轻度休闲'提升到'有深度的生活模拟'`
4. **获取封面图 URL**：按以下优先级搜索游戏封面图，写入基本信息表的 `封面图` 字段：
   - **优先**：Steam 游戏用 `https://steamcdn-a.akamaihd.net/steam/apps/{app_id}/header.jpg`（搜索游戏英文名获取 Steam App ID）
   - **备选**：用 websearch 搜索 `{游戏英文名} game header image official` 获取官方封面图 URL
   - **找不到**：留空，网站会用渐变色兜底
5. **写入文件**：`D:\mechanism-lib\games\[GameEnglishName].md`
   - 文件名使用游戏英文名 PascalCase（如 `Hades.md`、`GenshinImpact.md`）
4. **如果该游戏已有 Game Card**：合并更新，不覆盖已有信息

**产出**：`D:\mechanism-lib\games\[GameName].md` 游戏卡文件

#### Step 5c：生成 Mechanism Cards（机制卡）

从三份报告中筛选值得入库的机制，每个机制生成一张机制卡。

1. **筛选入库机制**（按 `references/entry-templates.md` 的入库筛选规则）：
   - **必须入库**：Phase 2 标为结构创新的机制、Phase 1 标为 HIGH 吸收优先级的机制
   - **选择性入库**：Phase 1 标为 MEDIUM 且在 Phase 3 中被识别为体验支柱支撑机制的
   - **不入库**：品类标配、包装层变化、信息不足、LOW 优先级

2. **确定前缀**：
   - `[Excellent]`：Phase 2 结构创新的机制
   - `[Qualify]`：Phase 1 HIGH 但非结构创新的机制、MEDIUM + 体验支柱支撑的机制

3. **读取入库模板**：打开 `references/entry-templates.md`，找到"二、Mechanism Card"章节的模板

4. **逐个机制填充**：
   - 按字段来源对照表，从 Phase 1-3 的产出中提取数据填入对应字段
   - **价值介绍**（通俗转译之后、机制说明之前）：
     - **机制亮点**：从 Phase 1 综合叙述 → Top 5 的"亮点"行直接取；非 Top 5 机制从 Phase 1 做法概述中提炼最突出的设计特点
     - **借鉴价值**：从 Phase 1 综合叙述 → Top 5 的"借鉴价值"行直接取；非 Top 5 机制从 Phase 1 吸收判断"理由"列提炼
     - **改变了什么**：合并 Phase 2 创新判据 → 结构性论证 + Phase 3 核心循环影响，改写为一段完整论述；Qualify 卡（无创新判据表）从 Phase 3 核心循环影响单独取
   - **机制说明**（合并了原机制拆解+设计意图+核心循环影响）：
     - **机制描述**：Phase 1 做法概述展开，用分条列表呈现关键参数
     - **触发条件**：Phase 1 做法概述中提取
     - **服务体验目标**：Phase 3 层次1 → 服务体验，用分条列表呈现
     - **核心受益者**：从服务体验中提炼一句话目标玩家画像（如"把游戏当日常生活一部分的玩家"）
     - **核心循环影响**：Phase 3 层次1 → 核心循环影响，用分条列表呈现关键变化
     - **参考视频**：搜索 `[游戏英文名] + [机制名] + gameplay/mechanic` 或 `[游戏中文名] + [机制名] + 玩法/机制/攻略`，优先 B站/YouTube，取 1-2 个链接
   - **代价与取舍**：
     - **解决痛点**：Phase 3 层次1 → 设计意图
     - **代价与取舍**：Phase 3 层次1 → 代价与取舍
     - **可能的风险预示**：Phase 1 综合叙述 → 风险提示中与该机制相关的条目
     - **不适用场景**：Phase 3 层次1 → 不适用场景（不适合的玩家类型和游戏品类，对照品类原型词典）
   - **关联机制卡**（合并了原协同关系+关联参考）：
     - **结构角色**：Phase 3 层次2 → 结构性脆弱点
     - **同游戏内**：Phase 3 协同关系表格 → 提取涉及该机制的条目，填入三列表格（关联机制卡/协同类型/协同效果）
     - **其他游戏**：基于标签相似度从已有机制库中筛选候选卡（品类+乐趣+痛点加权匹配），然后为每个候选生成一句参考原因（描述两个机制之间的互动/参考关系），填入两列表格（关联机制卡/参考原因）
   - **标签信息**：读取 `references/标签标准化参考.md`，按判定规则为每个机制打标签
   - **一句话解释**：1-2句话不超过60字，用冒号分「机制特征」和「产生效果」，不要展开长篇比喻

5. **写入文件**：`D:\mechanism-lib\mechanisms\[GameEnglishName]\[前缀][品类]-[GameEnglishName]-[模块名].md`
   - 如果目录不存在，先创建
   - **模块名命名**：通俗易懂，禁止堆叠专有名词，鼓励比喻式命名（详见 entry-templates.md 命名规范）
   - 品类缩写见模板定义中的品类缩写表

6. **回填 Game Card**：将所有生成的 Mechanism Card 文件名填入 Game Card 的"收录机制卡"章节

**产出**：`D:\mechanism-lib\mechanisms\[GameName]\` 目录下的若干机制卡文件

#### Step 5d：更新全局机制索引

**触发条件**：Step 5c 完成后自动执行

**做什么**：
将新生成的 Mechanism Cards 注册到全局索引 `knowledge/_index.md` 中。

1. **读取全局索引**：打开 `~/.codemaker/skills/mechanism-analyse/references/knowledge/_index.md`

2. **更新注册表(§1)**：为每张新生成的 Mechanism Card 追加 1 行
   - ID 从当前最大 ID 递增（如 M37、M38...）
   - 填入：机制卡名、游戏名、品类、级别、一句话解释、文件路径

3. **更新倒排索引(§2)**：从每张新卡的"标签信息"section 提取数据
   - §2.1 痛点：从"玩家痛点"标签提取，追加到对应痛点类别（每卡最多 3 条）
   - §2.2 体验：从"情绪反馈"+"16乐趣标签"提取，追加到对应体验类别（每卡最多 2 条）
   - §2.3 机制模式：从"特色机制"标签提取，归入最匹配的模式类别；如无匹配则新建类别
   - §2.4 约束：从"玩法层级"8 维提取，追加 ID 到对应分组
   - §2.5 协同：从"关联机制卡-同游戏内"的协同关系提取

4. **更新跨品类桥梁(§3)**：如果新卡的"其他游戏"关联指向不同品类的机制，考虑追加桥梁条目

5. **更新元信息**：修改文件顶部的"最后更新"日期、机制卡总数、游戏数（如果是新游戏）、品类数（如果是新品类）

**产出**：更新后的 `knowledge/_index.md`

#### Step 5e：生成 Gameplay Cards（玩法卡）

**触发条件**：Step 5d 完成后自动执行

**做什么**：
从 Phase 0 玩法拓扑报告 + Phase 1-3 的分析中提取数据，为每个玩法生成一张 Gameplay Card。

1. **读取入库模板**：打开 `references/entry-templates.md`，找到"四、Gameplay Card"章节的模板

2. **主玩法卡（始终生成）**：
   - 从 Phase 0 报告提取：核心循环、目标、约束、节奏
   - 从 Phase 1 提取：涉及的关键机制（指向机制卡）
   - 从 Phase 3 提取：体验画像、节奏曲线
   - 写入 `D:\mechanism-lib\gameplays\[GameEnglishName]\主玩法-[玩法名].md`

3. **副玩法卡（按 Phase 0 判定结果）**：
   - 从 Phase 0 报告提取：差分表、一句话定位、体验差异、设计意图
   - 从 Phase 1 提取：机制参数覆写（如有）
   - 填写"相对主玩法差分"section
   - 写入 `D:\mechanism-lib\gameplays\[GameEnglishName]\副玩法-[玩法名].md`

4. **回填 Game Card**：将所有生成的 Gameplay Card 文件名填入 Game Card 的"收录玩法卡"章节

**产出**：`D:\mechanism-lib\gameplays\[GameEnglishName]\` 目录下的玩法卡文件

**完成标志**：五类文件全部生成并更新——玩法卡包含规则集+差分、知识卡片包含精华摘要、Game Card 包含游戏级总览+收录玩法卡列表、Mechanism Cards 包含机制拆解和标签、全局索引包含新卡注册

## 四、执行节奏

### 核心原则：轻量交付

**只告诉我重要的，完整产出全部存文件。**

每个 Phase 完成后：
- ✅ **完整报告照常写文件**（reports/ 路径不变，用户可随时查看）
- ✅ **只发一条 TG 消息**，3~5 句话，告诉我最重要的结论
- ❌ **不再发完整报告全文**（除非用户主动要求）

### 执行记录（每个 Phase 完成后填写）

每次 Phase 开始前记录 `session_status`，结束后再记录一次，两者相减得到该 Phase 的消耗。

```
| Phase | 耗时 | Token消耗 | 交付消息 |
|--------|------|-----------|---------|
| Step 0 搜索 | — | — | 信源X个，覆盖XXX |
| Phase 0 玩法拓扑 | — | — | 主玩法X个，副玩法X个 |
| Phase 1 机制扫描 | — | — | X个系统，Y个通过初筛 |
| Phase 1.5 知识检索 | — | — | — |
| Phase 2 创新评价 | — | — | 结构创新X个，标配排除X个 |
| Phase 3 机制推理 | — | — | 体验支柱X个，承重墙X个 |
| Phase 5 标准化入库 | — | — | 入库X个（优秀X个，典型X个） |
| **合计** | **—** | **—** | |
```

**操作方式**：
1. 每个 Phase 开始前：调用 `session_status` 获取当前 token 数和 session 开始时间
2. Phase 结束后：再次调用 `session_status`，两次相减得到消耗
3. 填入上表，同时按下方"TG 交付模板"发一条消息
4. 全部完成后汇总合计行

**TG 交付模板**（每 Phase 只发一条）：

- "跳过评价，直接推理" → 允许跳过 Phase 2（reason 不强依赖 evaluate，但有 evaluate 的输入会更好）

## 六、禁止行为

- **禁止**跳过 Phase 1 直接做评价或推理（没有扫描报告，后续分析没有数据基础）
- **禁止**在 Phase 间重复搜索已有信息（后续 Phase 应复用前序产出）
- **禁止**自动触发 Phase 4（必须用户明确要求）
- **禁止**合并三份产出为一份（保持每份报告独立完整）
- **禁止**修改子 skill 的分析逻辑（编排 skill 只管调用顺序，不改分析方法）
- **禁止**将品类标配或包装层变化入库（只入库结构创新和承重墙机制）
- **禁止**覆盖已有知识卡片（合并追加，不覆盖）
- **禁止**覆盖已有 Game Card 或 Mechanism Card（合并更新，不覆盖）
- **禁止**生成 Mechanism Card 时跳过标签标注（必须按标签标准化参考.md 的判定规则打标签）
- **禁止**在 Mechanism Card 的一句话解释中使用术语（必须用生活/商业类比）
