# game-analyse - 游戏机制全流程分析

输入一个游戏名，自动串联三个 skill 完成从扫描到推理的完整分析。

**产出**：
1. 机制扫描报告（mechanism-analyse 模式一）
2. 创新评价卡片（mechanism-evaluate）
3. 机制推理报告（mechanism-reason）

可选：对特定机制做深度分析（mechanism-analyse 模式二）

---

## 一、使用方式

### 触发词
- "分析 [游戏名]"
- "拆解 [游戏名]"
- "看看 [游戏名]"
- "[游戏名] 怎么样"
- "研究一下 [游戏名]"
- "帮我分析 [游戏名]"

### 输入
- 游戏名称（必须）
- 项目背景（可选）——如"我在做一个搜打撤项目"，会影响 evaluate 的判断权重

---

## 二、执行流程

### Phase 1：机制扫描

**调用**：mechanism-analyse 模式一

**做什么**：
- 确定游戏品类，加载通用清单 + 品类修正补丁
- 搜索游戏信息（评测/Wiki/开发者访谈）
- 逐项扫描 57 个子模块，填写机制清单表格
- 输出综合叙述（机制特色 / Top 5 / 品类对比 / 风险提示）

**产出**：`[游戏名] 机制扫描报告`

**完成标志**：扫描报告包含完整的机制清单表格和综合叙述

**交接给下一步的数据**：整份扫描报告

---

### Phase 1.5：知识库检索

**触发条件**：Phase 1 完成后自动执行

**做什么**：
- 从扫描报告的"基础信息→品类"中提取品类关键词
- 扫描 `~/.codemaker/skills/mechanism-analyse/references/knowledge/` 目录，查找同品类已分析游戏的知识卡片
- 如果找到相关知识卡片，读取其中的"结构创新"和"体验支柱"作为后续 Phase 的参考基线
- 将已有知识简要提示给后续 Phase："同品类已分析过 [游戏A]、[游戏B]，它们的结构创新包括 [列表]"

**如果知识库为空**：跳过此步骤，继续 Phase 2

**价值**：
- 让 Phase 2（评价）在判断"品类标配 vs 结构创新"时有更多参考——如果同品类的其他游戏已经有同样的做法，更可能是品类标配
- 让 Phase 3（推理）在分析协同关系时可以对比"这个游戏的体验支柱和同品类其他游戏有什么不同"

---

### Phase 2：创新评价

**调用**：mechanism-evaluate

**输入**：Phase 1 产出的扫描报告

**做什么**：
- 读取扫描报告，做完整度检查
- 识别品类原型（对照品类原型词典）
- 三层拆解（原型层 / 创新层 / 包装层）
- 对候选结构创新逐条过 3 条硬性门槛
- 做品类标配排除
- 给出创新定性（换皮无创新 / 原型+结构创新 / 品类突破 / 伪创新）

**产出**：`[游戏名] 创新评价卡片`

**完成标志**：评价卡片包含品类原型判断、结构创新判据表格、创新定性

**交接给下一步的数据**：评价卡片 + 扫描报告（reason 需要两份）

---

### Phase 3：机制推理

**调用**：mechanism-reason

**输入**：Phase 1 的扫描报告 + Phase 2 的评价卡片

**做什么**：
- 从报告中筛选重点机制（HIGH 优先级 + 结构创新 + ~ 标记的偏离项）
- 层次1：对每个重点机制执行 5 步推理链（设计意图 / 作用机制 / 核心循环影响 / 服务体验 / 代价取舍）
- 层次2：识别体验支柱，绘制机制协同图（乘法/保险丝/弥补/链式关系），识别承重墙与装饰

**产出**：`[游戏名] 机制推理报告`

**完成标志**：推理报告包含层次1表格 + 层次2体验支柱和协同关系 + 设计精华总结

---

### Phase 4（可选）：单机制深挖

**触发条件**：用户在看完前三步产出后，指定某个机制要求深入分析

**调用**：mechanism-analyse 模式二（路径A：从扫描报告取数据）

**做什么**：
- 从扫描报告中提取目标机制的做法概述作为基础
- 补充搜索设计意图和玩家反馈
- 执行 6 步推理链（机制本身 / 直接效果 / 服务目标 / 核心循环影响 / 用户影响 / 代价取舍）

**产出**：`[游戏名] - [机制名] 深析报告`

**注意**：此步骤不自动触发，仅在用户明确要求时执行。

---

### Phase 5：知识入库

**触发条件**：Phase 3 完成后自动执行

**做什么**：
从三份报告中提炼高价值机制，写入知识库。

1. **筛选入库内容**：
   - 从 Phase 2（评价卡片）提取：所有被标为结构创新的机制（通过了 3 条门槛的）
   - 从 Phase 3（推理报告）提取：体验支柱、协同关系、承重墙机制、设计精华
   - 品类标配不入库、包装层变化不入库、信息不足的判断不入库

2. **生成知识卡片**：
   - 按 `~/.codemaker/skills/mechanism-analyse/references/knowledge/_template.md` 的格式生成
   - 文件路径：`~/.codemaker/skills/mechanism-analyse/references/knowledge/[游戏名英文小写].md`

3. **如果该游戏已有知识卡片**：
   - 读取已有卡片，合并新发现（不覆盖已有内容，追加新的结构创新）
   - 更新分析日期

**产出**：`knowledge/[游戏名].md` 知识卡片文件

**完成标志**：知识卡片包含品类原型、创新定性、结构创新列表、体验支柱表格、承重墙列表、设计精华

## 三、Phase 间的数据传递

```
Phase 1 (analyse)
  │
  ├─ 扫描报告 ──→ Phase 1.5 (知识检索) ──→ 已有知识作为参考
  │                                           │
  ├─ 扫描报告 ──┬──→ Phase 2 (evaluate) ←────┘
  │             │       │
  │             │       ├─ 评价卡片
  │             │       │
  ├─ 扫描报告 ──┤──→ Phase 3 (reason)
  │             │       │
  │  评价卡片 ──┘       ├─ 推理报告
  │                     │
  │  评价卡片 + 推理报告 ──→ Phase 5 (入库) ──→ knowledge/[游戏名].md
  │
  └─ 扫描报告 ──→ Phase 4 (analyse 模式二) [可选，用户触发]
```

**关键原则**：每个 Phase 的输入来自前序 Phase 的产出，不需要重新搜索已有的信息。

---

## 四、执行节奏

### 默认模式：分步执行，每步完成后交付

每完成一个 Phase，立即将产出展示给用户，然后继续下一步。不要等全部做完再一起给。

**节奏模板**：
1. "开始 Phase 1：扫描 [游戏名] 的机制..." → 执行 → 交付扫描报告
2. （Phase 1.5 自动执行，如有已知知识则简要提示："同品类已分析过 [游戏]，参考其结构创新..."）
3. "Phase 2：基于扫描报告评价创新层次..." → 执行 → 交付评价卡片
4. "Phase 3：推理机制间的因果和协同关系..." → 执行 → 交付推理报告
5. "Phase 5：提炼高价值机制入库..." → 执行 → 交付入库确认
6. "全流程分析完成。如需对某个具体机制深挖，告诉我机制名。"

### 中断与恢复

用户可以在任何 Phase 完成后暂停：
- "先到这里" → 停止，保留已有产出
- "继续" → 从下一个 Phase 继续
- "跳过评价，直接推理" → 允许跳过 Phase 2（reason 不强依赖 evaluate，但有 evaluate 的输入会更好）

---

## 五、语气与表达

### Phase 间的过渡
每个 Phase 开始前用一句话说明目的，不要铺垫：
- "扫描完成。接下来评价创新层次。"
- "评价完成。接下来推理机制间的协同关系。"

### 最终总结
全流程完成后，给一句话总结：
> "[游戏名] 的核心是一个 [品类原型] 产品，通过 [包装] 和 [N个结构创新] 形成差异化。它成立的关键在于 [设计精华]。已入库 [N] 条机制知识。"

---

## 六、禁止行为

- **禁止**跳过 Phase 1 直接做评价或推理（没有扫描报告，后续分析没有数据基础）
- **禁止**在 Phase 间重复搜索已有信息（后续 Phase 应复用前序产出）
- **禁止**自动触发 Phase 4（必须用户明确要求）
- **禁止**合并三份产出为一份（保持每份报告独立完整）
- **禁止**修改子 skill 的分析逻辑（编排 skill 只管调用顺序，不改分析方法）
- **禁止**将品类标配或包装层变化入库（只入库结构创新和承重墙机制）
- **禁止**覆盖已有知识卡片（合并追加，不覆盖）
