# gameplay-analyse - 游戏玩法拓扑分析

本 skill 用于识别一个游戏中有哪些可区分的玩法模式，判断主/副玩法，并分析副玩法相对主玩法的规则变异与体验差异。

---

## 一、核心概念

### 什么是"玩法（Gameplay）"

**玩法** = 游戏中一组可识别的、有独立操作/规则/产出的游玩活动。

它不是一个机制，也不是一个子系统，而是玩家能感知到的一种"可以做的事情"——有独立的动作、独立的产出、并且不做也行（副玩法）或者必须做（主玩法）。

### 三层区分

| 层级 | 定义 | 例子 |
|------|------|------|
| **主玩法** | 不玩它游戏根本无法推进的核心活动循环 | LoL 排位、星露谷种地、原神探索+主线 |
| **副玩法** | 不玩它主循环仍然成立（只是慢/少），有独立的操作方式或进度线 | LoL ARAM、星露谷钓鱼/烹饪/社交、原神深境螺旋 |
| **子系统/机制** | 没有独立的"玩的动作"，只是数据处理或支撑环节 | 背包整理、装备强化界面、技能加点 |

### 判定三条线

判断某个活动是"玩法"还是"子系统"，依次问三个问题：

| # | 判定问题 | 是 → | 否 → |
|---|---------|------|------|
| 1 | **独立性**：不碰这个活动，主循环是否还能转？ | 副玩法候选 | 主玩法组成部分 |
| 2 | **操作性**：这个活动是否有独立的"玩的动作"（不是纯点击确认）？ | 玩法 ✓ | 偏子系统 |
| 3 | **产出性**：是否有独立的产出/奖励/进度？ | 玩法 ✓ | 偏子系统 |

**规则**：
- 问题 1 回答"否" → 这个活动是主玩法的一部分（不一定单独出卡，但会在主玩法卡中描述）
- 问题 2+3 都回答"是" → 确认是副玩法，出卡
- 问题 2 或 3 只有一个"是" → 边界案例，记录但标注"弱副玩法"，由分析师判断是否出卡
- 问题 2+3 都回答"否" → 子系统/机制，不出玩法卡

### 主玩法判定评分（用于多主玩法候选时）

当游戏有多个活动都"不玩就推不动"时，用 5 维评分区分主次：

| 维度 | 0 分 | 1 分 | 2 分 |
|------|------|------|------|
| 入口地位 | 隐藏入口 | 可见但非默认 | 新手默认/教程承载/主菜单首按钮 |
| 时间份额 | <10% | 10-40% | >40% 活跃时长 |
| 经济闭环 | 无独立产出 | 部分独立 | 核心经济围绕此玩法 |
| 竞技/叙事主线 | 无 | 部分承载 | 排位/主线在此 |
| 常驻性 | 限时 | 周期常驻 | 永久常驻+优先平衡 |

最高分为主玩法。允许双主玩法（如 Destiny 2 的 PvE + PvP 都是核心入口且经济闭环独立）。

Phase 0（玩法拓扑扫描）→ Phase 1（机制扫描）→ Phase 1.5 → Phase 2 → Phase 3 → Phase 5

## 三、执行流程

### Step 1：信息收集

**在 game-analyse 全流程中**：使用 Phase 0+1 共享搜索的结果（Step 0），不再独立搜索。

**单独使用时**：执行以下搜索：
- 搜索游戏的官方介绍、Steam/App Store 页面、玩家评测
- 重点关注：游戏有哪些"可以做的事情"（活动列表）、哪些是核心循环的一部分、哪些是可选的
- 是否有独立的游戏模式/队列/入口
- 各活动的时间投入占比和玩家讨论热度

### Step 2：玩法识别

对每个候选活动，依次回答三条判定问题：

```
| 活动名 | ①独立性(不玩主循环转不转) | ②操作性(有独立动作吗) | ③产出性(有独立产出吗) | 判定 |
|--------|---|---|---|---|
| [活动A] | 不转 → 主玩法部分 | - | - | 主玩法组成 |
| [活动B] | 转 | ✓ 有独立操作 | ✓ 有独立产出 | 副玩法 ✓ |
| [活动C] | 转 | ✗ 纯确认 | ✓ 有产出 | 边界(弱副玩法) |
```

### Step 3：主玩法基线定义

从 Step 2 中"主玩法组成"的活动出发，综合定义主玩法：

1. **主玩法名称**：用一个可理解的名字概括核心循环（如"种田经营循环"、"5v5 召唤师峡谷"、"开放世界探索+主线"）
2. **核心循环描述**：[起点] → [核心动作] → [产出/结算] → [回到起点]
3. **核心目标**：玩家在这个玩法中追求什么
4. **关键约束**：什么规则限制了自由度（时间/资源/空间/对手）
5. **典型节奏**：一次完整循环的时长和节奏特征

### Step 4：副玩法分类与分析

#### Step 4a：判断副玩法与主玩法的关系

每个副玩法先判断它和主玩法的关系类型：

| 关系类型 | 判定标准 | 分析方式 | 一句话定位格式 |
|---------|---------|---------|-------------|
| **主玩法变体** | 核心操作逻辑和主玩法相同，只是改变了部分规则/约束/目标 | 写差分表（改了什么→体验怎么变） | "在[主玩法名]的基础上，通过[改了什么]，提供[什么体验]" |
| **独立玩法** | 操作逻辑、对抗结构、或核心循环和主玩法有本质区别 | 写独立定位（它是什么）+ 共享资产（和主玩法共享什么） | "[直接描述这个玩法是什么、提供什么体验]" |

**判定方法**：问一个问题——"如果把主玩法完全删掉，这个副玩法的规则还能独立成立吗？"
- **能成立** → 独立玩法（如 TDM 不依赖猎场的 PVE 闯关逻辑）
- **不能成立** → 主玩法变体（如 竞速模式 = 猎场的计时版本，没有猎场地图就没有竞速）

#### Step 4b：主玩法变体 → 写差分

```
### [副玩法名称]（主玩法变体）

**一句话定位**：在[主玩法名]的基础上，通过 [改了什么]，提供 [什么不同的体验]。

**规则差分**：

| 维度 | 主玩法 | 本玩法 | 变更方向 |
|------|--------|--------|---------|
| （仅列出有变化的维度，不强制填满 7 行） |

**体验差异**（3 句以内）：
> [这个玩法和主玩法玩起来感觉有什么不同]

**设计意图**（1-2 句）：
> [为什么做这个副玩法]
```

#### Step 4c：独立玩法 → 写独立定位

```
### [副玩法名称]（独立玩法）

**一句话定位**：[直接描述这个玩法是什么、核心体验是什么，不强行锚定主玩法]

**核心规则**：
- 核心目标：[目标]
- 操作方式：[操作]
- 时间结构：[时长]
- 社交结构：[人数/对抗/合作]

**共享资产**（和主玩法/其他玩法共享什么）：

| 共享内容 | 说明 |
|---------|------|
| [如：角色/装备/货币/天赋] | [怎么共享：全共享/部分/独立] |

**体验差异**（3 句以内）：
> [相对游戏整体，这个玩法提供什么独特价值]

**设计意图**（1-2 句）：
> [为什么做这个玩法]
```

## 四、与后续 Phase 的衔接

### 给 Phase 1（机制扫描）的输入

Phase 0 完成后，向 Phase 1 传递：
- 主玩法名称和核心循环 → Phase 1 的扫描以主玩法为基准
- 副玩法列表 → Phase 1 扫描时标注"此机制在哪些玩法中出现/参数是否不同"
- 如果副玩法数量 ≥ 3 且差异大 → 提示 Phase 1 按玩法分组扫描

### 给 Phase 5（入库）的输入

Phase 0 的报告 + Phase 1-3 的分析 → Phase 5 Step 5e 生成 Gameplay Card：
- 主玩法 → 始终生成 1 张卡
- 副玩法 → 每个确认的副玩法生成 1 张卡（边界弱副玩法可选）
- 单玩法游戏（如只有种地主循环 + 无明显副玩法）→ 仅生成主玩法卡

---

## 五、边界约束

### 限时活动处理
- 如果限时活动的**玩法框架可复用**（如"限时 PvP 竞技场"反复开放）→ 出 1 张卡标注"限时-周期复现"
- 如果每期活动规则都不同且不复现 → 不出卡，在主玩法卡中注记"有限时活动轮换"

### "分家"判定
当副玩法满足以下全部条件时，应升级为独立游戏卡而非玩法卡：
- 拥有独立的长期经济闭环
- 拥有独立的内容更新管线
- 拥有独立的社区和竞技体系
- 示例：云顶之弈从 LoL 副玩法 → 独立游戏

### 单玩法游戏
如果 Phase 0 判定游戏只有 1 个主玩法 + 0 个副玩法 → 仍然输出主玩法卡（描述核心循环），但跳过差分分析。
