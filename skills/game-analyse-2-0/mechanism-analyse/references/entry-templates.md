# 入库模板定义

本文件定义 game-analyse 全流程分析完成后，Phase 5 自动生成的两种入库卡片格式。

所有字段**均来自 Phase 1-3 的产出**，不需要额外搜索。本文件同时说明每个字段的数据来源。

## 二、Mechanism Card（机制卡）

机制级详细卡。每个值得入库的机制一张。

### 文件位置

`D:\mechanism-lib\mechanisms\[GameEnglishName]\[前缀][品类]-[GameEnglishName]-[模块名].md`

### 文件命名规则

**前缀**（根据入库筛选结果）：
- `[Excellent]` — Phase 2 标为结构创新 且 Phase 3 标为承重墙
- `[Qualify]` — Phase 2 标为结构创新 或 Phase 1 标为 HIGH 吸收优先级

**品类缩写**：

| 品类原型 | 缩写 |
| :--- | :--- |
| COK-like / 率土-like / 4X-like | SLG |
| 卡牌RPG-like / 回合制RPG-like | RPG |
| ARPG-like | ARPG |
| 传统MMO-like / 轻量MMO-like | MMO |
| 经典放置-like / 放置+X混合型 | 放置 |
| BR-like | BR |
| 战术射击-like | FPS |
| MOBA-like | MOBA |
| 农场经营-like / 城市建造-like | 模拟经营 |
| 生存建造-like | 生存 |
| 经典Roguelike-like | Roguelike |
| 经典塔防-like | 塔防 |
| 派对竞技-like | 派对 |
| 三消-like / 混合休闲-like | 休闲 |

品类缩写从 Phase 2 的品类原型判断中提取。如果是混合型（A+B），取主要品类。

**模块名**：从 Phase 1 扫描报告的子模块名或 Phase 2 的创新机制名提取，使用中文，简洁概括。

**模块名命名规范**：
- **通俗易懂**：读者无需游戏设计背景也能大致猜到这个机制是什么
- **禁止堆叠专有名词**：不要用"跨系统元目标"、"FOMO"等需要额外解释的术语
- **鼓励比喻式命名**：用类比让人秒懂——"集章卡式收集"比"跨系统元目标"好；"无压力日常设计"比"无FOMO日循环"好
- **点明机制作用**：命名应暗示这个机制"做了什么"，而非它"叫什么"
- ✅ 好例子：大头菜社交经济、集章卡式收集系统、无压力日常设计、帕鲁劳工系统
- ❌ 坏例子：无FOMO日循环、社区中心跨系统元目标、NPC独立日程社交

**示例**：
```
[Excellent]Roguelike-Hades-叙事绑定死亡.md
[Qualify]Roguelike-Hades-神明祝福构筑.md
[Excellent]RPG-GenshinImpact-元素反应系统.md
```

> **写作规范**：
> - **长度限制**：1-2 句话，不超过 60 字

## 三、入库筛选规则

### 什么机制生成 Mechanism Card

从 Phase 1 扫描报告的 57 子模块中筛选：

**必须入库**（自动生成 Mechanism Card）：
1. Phase 2 标为**结构创新**（通过了 3 条门槛）的机制
2. Phase 1 标为**值得吸收 = 是 且 优先级 = HIGH** 的子模块

**选择性入库**（生成但标记为 Qualify）：
3. Phase 1 标为**值得吸收 = 是 且 优先级 = MEDIUM** 的子模块，且在 Phase 3 的推理中被识别为体验支柱的支撑机制

**不入库**：
- 品类标配（Phase 2 排除的机制）
- 纯包装层变化
- 信息不足（Phase 1 标记为"信息不足"的子模块）
- 优先级为 LOW 的子模块
- 不存在（✗）或不适用（N/A）的子模块

### 前缀判定

| 条件 | 前缀 |
|------|------|
| Phase 2 结构创新 + Phase 3 承重墙 | `[Excellent]` |
| Phase 2 结构创新 + Phase 3 非承重墙 | `[Excellent]` |
| Phase 1 HIGH + 非结构创新 | `[Qualify]` |
| Phase 1 MEDIUM + 体验支柱支撑 | `[Qualify]` |

### Game Card 始终生成

每次全流程分析完成后，无条件生成一张 Game Card。如果该游戏已有 Game Card，合并更新（不覆盖已有信息，追加新分析）。

### Gameplay Card 按 Phase 0 结果生成

- 主玩法 → 始终生成 1 张卡
- 副玩法 → 每个确认的副玩法生成 1 张卡
- 边界弱副玩法 → 由分析师判断是否出卡
- 单玩法游戏（无副玩法）→ 仅生成主玩法卡
