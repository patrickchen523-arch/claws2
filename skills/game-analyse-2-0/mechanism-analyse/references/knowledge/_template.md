# 知识卡片格式规范

本文件定义机制知识库中每个游戏知识文件的标准格式。

## 目录结构

知识库按品类分文件夹存储：

```
knowledge/
├── _index.md             # 全局机制索引（跨品类倒排索引，Phase 1.5 + V3 提问流使用）
├── _template.md              # 本文件
├── farming-sim/              # 农场经营
│   ├── _summary.md           # 品类知识总结（每次新增游戏后更新）
│   ├── stardew-valley.md
│   └── ...
├── life-sim/                 # 生活模拟
├── city-builder/             # 城建
├── automation/               # 自动化
├── survival/                 # 生存
├── match3/                   # 三消
├── tycoon/                   # 主题经营
├── extraction-shooter/       # 搜打撤
└── [new-genre]/              # 新品类按需创建
```

### 全局索引 `_index.md`

机制级倒排索引，按 5 个维度（痛点/体验/机制模式/约束/协同）索引所有 Mechanism Card。

**用途**：
- Phase 1.5 Step 1.5a：跨品类检索与当前游戏相关的机制卡
- V3 提问流（预留）：根据用户描述推荐最匹配的机制卡

**维护**：Phase 5 Step 5d 自动更新（新增机制卡时追加注册行+索引条目）

## 品类文件夹命名规则

使用英文小写 + 连字符，对应品类原型词典中的品类。新品类按需创建文件夹。

| 文件夹名 | 品类中文 | 示例游戏 |
|---------|---------|---------|
| farming-sim | 农场经营 | Stardew Valley, Hay Day |
| life-sim | 生活模拟 | Animal Crossing, Pokemon Pokopia |
| city-builder | 城建 | Cities Skylines, SimCity |
| automation | 自动化 | Factorio, Dyson Sphere |
| survival | 生存 | Palworld |
| match3 | 三消 | Homescapes |
| tycoon | 主题经营 | Two Point Campus |
| extraction-shooter | 搜打撤 | Marathon |

## 文件命名

- 知识卡路径：`knowledge/[品类文件夹]/[游戏名英文小写].md`
- 示例：`knowledge/farming-sim/stardew-valley.md`
- 品类总结：`knowledge/[品类文件夹]/_summary.md`

## 文件结构

```markdown
# [游戏名] 机制知识卡

**品类原型**：[原型名，如 经典Roguelike-like]
**创新定性**：[换皮无创新 / 原型+结构创新 / 品类突破 / 伪创新]
**分析日期**：[YYYY-MM-DD]

---

## 结构创新

### [创新名称]
- **来源子模块**：[编号+名称]
- **做法**：[一句话概述具体设计]
- **解决的问题**：[这个设计解决了什么痛点或创造了什么体验]
- **协同关系**：[与哪些其他机制形成什么类型的协同（乘法/保险丝/弥补/链式）]
- **移植注意**：[借鉴这个设计时需要注意什么条件或风险]

(每个结构创新重复以上格式)

---

## 设计精华

> [一句话总结这个游戏最精妙的机制协同关系]

## 体验支柱

| 支柱 | 支撑机制 | 核心协同 |
|------|---------|---------|
| [支柱名] | [机制A + 机制B + ...] | [协同类型和效果] |

## 承重墙机制

| 机制 | 拿掉后的影响 |
|------|------------|
| [机制名] | [哪些支柱会崩] |
```

## 入库规则

### 什么该入库
- 被 mechanism-evaluate 标为结构创新（通过 3 条门槛）的机制
- 被 mechanism-reason 识别为承重墙的机制
- 在体验支柱中处于乘法关系核心位置的机制

### 什么不该入库
- 品类标配（已在品类原型词典中的标志特征）
- 纯包装层变化（题材/美术/UI，不涉及结构）
- 信息不足、置信度为低的判断

### 入库质量标准
- 每个结构创新必须有具体做法描述（不能只写"有创新"）
- 必须写明解决的问题（不能只写"体验好"）
- 协同关系必须指明具体的协同类型（不能只写"有关联"）
