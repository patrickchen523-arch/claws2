# mechanism-reason - 游戏机制推理

本 skill 基于 mechanism-analyse 的扫描报告，做两个层次的推理：
- **层次1：单机制推理** —— 每个重要机制为什么存在？它服务于游戏的什么目标？
- **层次2：机制协同推理** —— 这些机制怎么共同支撑游戏成立？它们之间的联动关系是什么？

**核心价值**：不是描述"有什么"（那是 mechanism-analyse 的活），而是解释"为什么这么设计"和"拿掉一个会怎样"。

---

## 一、使用方式

### 触发词
- "单独推理 [游戏名] 的机制逻辑"
- "只做机制推理"
- "单独分析机制协同"
- "这些机制怎么配合的"（已有报告上下文时）

### 输入要求
- **默认输入**：对话上下文中的 mechanism-analyse 模式一扫描报告
- **指定文件**：用户可提供报告文件路径
- **可选输入**：mechanism-evaluate 的评价卡片（如有，可直接使用其基酒/气泡判断作为推理起点）

### 两种使用模式
- **全局推理**："推理 Hades 的机制逻辑" → 同时输出层次1 + 层次2
- **定向推理**："Boon系统在Hades中起什么作用" → 仅输出该机制的层次1，并展示它与其他机制的关联

---

## 二、执行流程

### Step 1: 读取报告并识别重点机制

从 mechanism-analyse 扫描报告中提取需要推理的机制：

**重点机制筛选**（全局推理时）：
1. **综合叙述 → Top 5 值得吸收的机制**：这些是报告作者认为最重要的，全部纳入
2. **机制清单中优先级为 HIGH 的子模块**：逐个纳入
3. **标记为 ~ 且做法概述显示明显偏离品类常规的子模块**：纳入
4. **综合叙述 → 机制特色中提到的机制**：交叉确认是否已纳入

通常会得到 5-8 个重点机制。不需要对所有 57 个子模块都做推理——只对"让这个游戏成为它自己"的关键机制做推理。

---

### Step 2: 层次1 —— 单机制推理

对每个重点机制，执行 5 步推理链。数据来源以扫描报告中的"做法概述"和"理由"列为主，补充搜索为辅。

**5 步推理链**：

**① 设计意图**：这个机制是为了解决什么问题而存在的？
- 从报告的"理由"列提取痛点/体验线索
- 问自己：如果没有这个机制，玩家会碰到什么问题？
- 产出："这个机制解决的是 ______ 问题"

**② 作用机制**：它通过什么方式起作用？
- 从"做法概述"中提取具体设计参数和规则
- 分析因果链：玩家做了 X → 触发了 Y → 感受到了 Z
- 产出："它通过 ______ 机制，让玩家产生 ______ 体验"

**③ 对核心循环的影响**：它怎么改变了"玩一局游戏"的基本流程？
- 核心循环的哪个环节被这个机制影响？
- 影响的是节奏（快慢）、结构（流程顺序）还是深度（决策空间）？
- 产出："它作用于核心循环的 ______ 环节，带来的变化是 ______"

**④ 服务的体验目标**：它最终服务于什么样的玩家体验？
- 参考体验维度：16 乐趣标签、8 情绪反馈、痛点解决
- 这个机制主要服务于哪类玩家（硬核/大众/休闲）？
- 产出："最终服务于 ______ 体验，核心受益者是 ______ 玩家"

**⑤ 代价与取舍**：为了实现这个设计，牺牲了什么？
- 从报告的"风险提示"中提取相关风险
- 思考：这个机制在什么情况下会失效或产生负面效果？
- 产出："代价是 ______，风险在于 ______"

**⑥ 不适用场景**：这个机制不适合什么玩家和什么类型的游戏？
- 从⑤的代价与取舍反推：哪些玩家会因为这个代价而流失？
- 对照品类原型词典，思考：哪些品类的核心诉求与这个机制冲突？
- 产出："不适合 ______ 类型玩家（因为 ______）；不适合 ______ 类型游戏（因为 ______）"

---

### Step 3: 层次2 —— 机制协同推理

层次1完成后，进入协同分析。这是本 skill 的核心价值——不是一个个机制孤立看，而是看它们怎么"配合"。

#### 3a. 识别体验支柱

从层次1的推理结果中，提炼这个游戏的 2-4 个核心体验支柱。

体验支柱 = "玩家为什么持续玩这个游戏"的核心理由。每个支柱通常由 2-4 个机制协同支撑。

提炼方法：
- 将层次1中每个机制的"服务体验目标"归类
- 多个机制服务于同一个体验目标 → 这就是一个支柱
- 用玩家视角命名（如"每局都不一样"而非"Build多样性系统"）

#### 3b. 绘制机制协同图

对每个体验支柱，说明其下的机制如何协同工作：

**协同关系类型**：
- **乘法关系**：机制A × 机制B 产生的体验远大于各自单独存在。例：Boon随机 × 固定操作位 = "每局手感熟悉但能力不同"
- **保险丝关系**：机制A 让玩家敢去体验机制B。例：God Mode 让休闲玩家敢去体验高难度Build
- **弥补关系**：机制A 解决了机制B 带来的副作用。例：叙事绑定死亡弥补了永久死亡的挫败感
- **链式关系**：机制A 的产出是机制B 的输入。例：Boss击杀→掉落 Titan Blood→解锁 Aspect→改变Build策略

对每对协同关系，写清楚：
- 哪两个机制在协同
- 协同类型是什么（乘法/保险丝/弥补/链式）
- 协同产生的具体效果是什么

#### 3c. 识别结构性脆弱点

推理"拿掉哪个机制，整个体验会崩"：
- 哪些机制是"承重墙"？（拿掉后多个支柱崩塌）
- 哪些机制是"装饰"？（拿掉后体验基本不变）
- 哪些机制之间的协同关系最密切——这些就是这个游戏的"设计精华"

---

### Step 4: 补充搜索（最多 3 次）

仅在以下情况触发：
- 报告中"做法概述"信息不足以判断设计意图
- 需要确认某个机制的影响范围（是局部还是全局）
- 需要开发者访谈/GDC 来确认设计意图

搜索关键词模板：
1. `[游戏名] + [机制名] + design intent / GDC / postmortem`
2. `[游戏名] + why [机制] + developer interview`
3. `[游戏名] + [机制A] + [机制B] + synergy / 协同`

---

### Step 5: 输出推理报告

使用第三节的输出模板。

## 四、语气与表达

### 开场
先给一句核心判断（"这个游戏成立的核心原因是 ______"），再展开推理链。

### 语气
直接、因果分明、可用于设计讨论会。

**优先用**：
- 本质上、真正的作用是、去掉这个会怎样
- 不在于……而在于……
- A 和 B 是乘法关系 / A 是 B 的保险丝
- 这是承重墙 / 这是装饰

**不要用**：
- 赋能、抓手、闭环、协同效应（用具体的协同类型名称替代）
- 笼统的"设计很巧妙"（要说清巧在哪）

### 表达层次
- 层次1用表格紧凑呈现
- 层次2用支柱结构 + 协同关系表格
- 关键判断用 **加粗** 标出
- 设计精华用引用块

## 五、禁止行为

- **禁止**自行扫描游戏机制（这是 mechanism-analyse 的工作）
- **禁止**捏造设计意图（必须基于报告数据或搜索证据推理）
- **禁止**对所有 57 个子模块都做推理（只对重点机制）
- **禁止**做创新判断（那是 mechanism-evaluate 的工作）
- **禁止**笼统地说"设计很好""很巧妙"——必须说清具体的因果关系和协同类型
- **禁止**输出 HTML 格式（纯 Markdown）
- **禁止**添加商业分析、付费策略建议
