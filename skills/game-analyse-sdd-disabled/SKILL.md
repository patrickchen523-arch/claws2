# Game-Analyse-SDD - 游戏分析标准化拆解

本技能严格按照提供的标准化拆解文档框架对游戏进行分析。

## 核心规则

- 结构化对齐：必须严格遵循本框架进行输出，不乱加模块，不漏掉关键项
- 聚焦使游戏成立的核心支柱（离开该系统游戏体验大打折扣）
- 观点来源：可借鉴网络上已有的分析，但需注明来源
- 系统性分析：游戏是整体，分析单一系统时必须结合相关系统一起看

## 执行流程

### 第一步：收集游戏基础信息

搜索整理游戏的开发者、开发者背景、上线时间、销量/收入等信息。

- 通过 wiki 或公开搜索引擎获取基本信息
- 通过公开新闻获取营收数据（若无法获取确切数据，需注明"缺乏确切数据支持"）
- 若只能搜到泛泛的"表现良好"，使用"据公开新闻显示……"或直接标明"缺乏确切数据支持"，严禁捏造具体营收数字

**输出格式参考：** "《XXX》是由XX公司自主研发的一款XX游戏，XX公司是XX领域的公司，代表作为《XX》等，《XX》在20XX年X月X日首次发布，上线首年收入XX亿元，最近一年收入XX亿元……"

### 第二步：分析游戏X-Statement公式、玩法原型

搜索以下内容：
1. 网络上已有的对该游戏的拆解、测评文章
2. 官网对于该游戏玩法的介绍
3. 用户对于该游戏讨论热度较高的内容
4. **该游戏的设计参考/借鉴了哪些已存在的游戏**

**关于「玩法原型」的深度分析方法：**

玩法原型不仅需要指出借鉴了哪个游戏，更要描述：
1. **被借鉴游戏的核心理念/体验是什么**（基于什么心理体验设计）
2. **本游戏在此基础上改出了什么独特的体验**

参考案例：
- 《超自然行动组》的玩法原型是《逃离塔科夫》
  - 《逃离塔科夫》的核心体验是**基于赌场一夜暴富的体验**：把筹码代入赌场（战备），选择牌桌体验（地图），可能暴富、小赚，也可能血本无归
  - 《超自然行动组》的独特体验是**四人合作闯荡鬼屋的刺激体验**：与队友一起进入鬼屋（地图），期间牵手探索并遭遇各类鬼怪惊吓，最后成功逃出鬼屋（撤离）

整理获取【游戏基础信息】、【所属品类】、【玩法原型】，并通过逻辑推理得出游戏的【核心差异点/独特卖点】。

**输出格式参考（X-Statement公式）：**
"【基础信息】+【品类】+【核心差异点/独特卖点】"
- 基础信息需包含：游玩人数、玩法规则(PvE/PvP等)、核心差异点/卖点

**输出格式参考（玩法原型）：**
"玩法原型：《XX》是基于YYY的体验：……"
- 玩法原型指**该游戏的设计参考/借鉴了哪个已存在的游戏**
- 如果游戏包含多个核心系统，需分别说明每个系统借鉴了哪些游戏
- 参考案例：
  - 蛋仔派对 → 借鉴《Fall Guys》糖豆人的多人闯关派对玩法
  - 明日方舟终末地 → 基建生产借鉴《戴森球计划》/《幸福工厂》的自动化建造，战斗系统借鉴《异度之刃》的即时战斗+连击系统
  - 喵喵的结合 → 战斗系统借鉴《以撒的结合》的Roguelite随机地牢，遗传育种借鉴宠物游戏的多代际养成
  - Marathon → 借鉴《Escape from Tarkov》的硬核撤离玩法，独特卖点是"酸性科幻美术 + RPG三层构筑"
  - 命运2 → 借鉴《无主之地》的刷宝射击，独特卖点是"Raid团队协作 + 8年科幻叙事 + 赛季运营"

### 第三步：拆解游戏支柱

核心支柱旨在明确产品的核心竞争力，80%的资源和体验集中投入到哪3-4个关键支柱上。

**每个支柱的输出格式：**
```
支柱X：【支柱体验】-> 【设计手段】-> 【具体参数】
```

**参数类型：**
- 数字化：具体的数值、百分比等量化参数
- 规则化：触发条件 + 响应行为
- 集合化：通过列举潜在的实现手段/元素来表达

**常见支柱类型参考：**
- 涌现式交互设计：环境元素（水、火、电）可产生传导、烹饪等连锁反应
- 代际传承机制：战利品/突变可遗传给后代，形成长线养成动力
- 资源循环系统：产出→消耗→再产出的闭环

**注意事项：**
1. 每个支柱要有明确的实现参数（Spec），不能只停留在形容词层面
2. 游戏支柱要聚焦于用户体验层面
3. 避免冗长段落，选用精简的高度概括短语

### 第四步：拆解核心体验

基于前文分析的逻辑总结，产出以秒为单位、以分钟/小时为单位的核心体验。

**以秒为单位的核心体验格式：**
【游戏镜头视角】+【玩家行为】+【乐趣】
参考案例："3D第三人称视角为主，玩家扮演探险家在随机生成的地图中寻宝，躲避怪物并撤离，享受微恐下的刺激摸金"

**以分钟/小时为单位的游戏节奏：**
根据游戏类型选择合适的描述方式：

**方式一：一局完整流程（适用于单局体验明确的游戏）**
描述一次完整游戏流程中，玩家自然经历的体验阶段。

参考案例：
- MOBA游戏："单局分为对线期（前10分钟）、资源团战期（10-20分钟）、后期团战期（20分钟+），每个阶段体验重点不同"
- PUBG："单局分为落地发育期（0-5分钟）、跑圈转移期（5-15分钟）、决赛圈打架期（15-25分钟）"
- 基建向游戏："探索收集期（在关卡中战斗探索）→休整建设期（返回基地消耗材料升级产线）→挑战推进期（带着更强装备进入新区域）"

**方式二：长线养成循环（适用于短单局 + 长线养成的游戏）**
描述以天/周为单位的日常循环节奏。

参考案例：
- 手游日常循环："每日日常（刷御魂、探索副本）→周常任务（阴阳寮、斗技挑战）→活动周期（大型版本活动、抽卡UP）"
- 代际传承循环（喵喵的结合）："房屋培育阶段（交配繁殖）→战斗准备阶段（选猫+配职业）→回合制战斗（10x10战场）→基因传递（存活猫带突变回家）→循环往复"

### 第五步：Steam 数据获取（如游戏在 Steam 上）

如需获取 Steam 游戏的量化数据（玩家活跃、评价），可按以下方式：

**数据来源：**
1. **SteamDB** - 用户活跃数据（峰值在线、销量估算）
   - 网址：`https://steamdb.info/app/{game_id}/charts/`
   - 注意：Cloudflare 防护较强，Playwright 可能需要更长等待时间

2. **SteamCharts** - 历史玩家数据
   - 网址：`https://steamcharts.com/app/{game_id}`

3. **Steam 商店** - 评价、售价
   - 网址：`https://store.steampowered.com/app/{game_id}/`
   - 注意：需要登录才能获取完整数据

4. **搜索引擎** - 新闻报道中的数据
   - 搜索 `game name Steam concurrent players peak 2025` 获取媒体报道

**示例（Marathon）：**
- 搜索 "Marathon Steam concurrent players peak" → Eurogamer 报道 143,621 人
- 对应 SteamDB App ID: 3065800

### 第六步：析出游戏机制清单

在完成游戏整体分析后，需要析出该游戏的具体机制清单，并进行价值筛选。

#### 6.1 析出候选机制

思考"这个游戏有哪些机制"，列出所有值得拆解的机制。

**常见机制分类参考：**
- **战斗类**：技能系统、武器系统、元素反应、闪避/格挡、连招系统...
- **经济类**：抽卡系统、体力系统、装备强化、货币系统、交易系统...
- **成长类**：天赋树、等级系统、装备系统、BD构筑...
- **社交类**：公会系统、好友系统、师徒系统、匹配系统...
- **内容类**：地图探索、任务系统、解谜关卡、剧情演出...

#### 6.2 剔除常见通用机制

建立"常识库"，以下机制通常不值得单独拆解：
- 移动、跳跃、基础背包
- 好友系统、常规匹配系统
- 签到系统、每日任务（除非有创新）
- 常规商城、常规抽卡（除非有保底等创新）

**如果选了一个机制，必须回答：**
> "这个机制在 [竞品A] 和 [竞品B] 中没有出现，或者做得更好的地方在哪里？"

#### 6.3 价值判定三原则

**创新度（Innovation）**：该机制是否打破了品类常识？
- 例：在射击游戏中加入"撤离"机制

**耦合价值（Synergy）**：该机制是否是多个系统的枢纽？
- 例：抽到的卡还能作为升级其他装备的耗材

**设计痛点解决（Problem Solving）**：该机制是否解决了品类顽疾？
- 例：用保底机制解决非酋挫败感

#### 6.4 "机制的灵魂"分析

对通过筛选的机制，撰写以下字段：

| 字段 | 说明 |
|------|------|
| **设计意图** | 这个机制是为了让玩家产生什么情绪？（紧张？爽快？） |
| **爽点/痛点平衡** | 玩家得到了什么，又付出了什么？ |
| **失效条件** | 在什么情况下，这个机制会变得极其无聊或破坏平衡？ |

#### 6.5 原子化 vs 场景化

- **原子化**：析出最小逻辑单元
  - 例："非对称视野锁定机制"
- **场景化**：析出多个机制组成的"Combo"
  - 例："深夜时段+低可见度环境+声音探测UI"构成的"生存压迫感组合拳"

#### 输出格式

```
## 机制清单

### 通过筛选的机制

| 机制名 | 差异化亮点 | 设计意图 | 爽点/痛点平衡 | 失效条件 |
|--------|-----------|----------|---------------|----------|
| 撤离机制 | 搜打撤独有，风险收益绑定 | 紧张感 | 高风险高回报/死亡全失 | 撤离点过多 |
| ... | ... | ... | ... | ... |

### 已剔除的常见机制

- 移动/跳跃系统
- 常规匹配系统
- ...
```

## 禁止行为

- 禁止把重点放在边缘的周边系统（如普通七日签到）
- 禁止"幻觉"与捏造：遇到没有资料的游戏机制，必须坦白说明"缺少相关信息"
- 禁止捏造没有信息来源的数据，如实说明"无法查找到相关数据"
- 禁止将常见通用机制入库（必须经过价值判定筛选）
