# 游戏机制条目生成器

将游戏拆解分析结果格式化为符合 jizhi 机制库标准的 Markdown 条目。

## 适用场景

- game-analyse-sdd 拆解完成后，需要生成入库格式
- 用户确认哪些支柱可以入库后，生成最终条目

---

## 机制库：设计详情标准卡 (Mechanism Card)

*核心资产库，采用高颗粒度拆解，确保不仅"能看懂"，还能"落地参考"。*

### 模板

```markdown
# [品类]-[游戏名]-[具体模块]

## 基础信息

| 字段 | 内容 |
| :--- | :--- |
| 案例名称 | [品类]-[游戏名]-[具体模块] |
| 来源游戏 | 🔗 *游戏名* |
| 核心标签 | 标准化标签（形式+体验） |
| 来源 | 人工 / AI生成-已甄别 / AI生成-未甄别 |

---

## 通俗转译

**一句话解释**：
> 用生活/商业类比，供非策划人员快速理解

---

## 机制拆解

**机制描述**：
> 详细的规则说明，包含交互流程

**触发条件**：
> 机制生效的时机或前置条件

---

## 设计意图

**解决痛点**：
> 该机制是为了解决什么负面体验？

**体验目标**：
> 该机制希望提供什么正面情绪？

---

## 关联参考

**相似机制**：
> 跨游戏的同类案例，方便横向对比

---

## 多媒体

**演示素材**：
> GIF动图或视频链接（必填）
```

## 游戏库：背景扫盲卡 (Context Card)

*辅助库，重点在于"扫盲"和"对标"，让不了解该游戏的人也能看懂机制的生效环境。*

### 模板

```markdown
# 游戏名 / Game Name

## 基本信息

| 字段 | 内容 |
| :--- | :--- |
| 游戏名称 | 游戏官方名称（中文/英文） |
| 一句话对标 | 核心字段。用已知游戏进行类比（如：带枪的DOTA2） |
| 游戏类型 | 基础分类（如：TPS / MOBA / 卡牌） |

---

## 玩法环境

**核心循环**：
> 简述单局怎么玩（如：搜集->跑圈->对枪）

**操作门槛**：
> 评估受众群体的上手难度

---

## 关联资产

**收录机制**：
> [自动关联] 显示机制库中所有属于该游戏的条目
```

### 字段规范

| 字段模块 | 字段名称 | 说明与填写规范 |

## 使用流程

### 步骤 1：接收任务
接收 game-analyse-sdd 的分析结果，确认要入库的机制和游戏

### 步骤 2：检查游戏卡是否存在
**重要**：在入库前，必须检查目标游戏是否已有游戏卡

1. **查找游戏库目录**：`/root/.openclaw/agents/jizhi/workspace/games/`
2. **检查方法**：
   - 如果游戏库中已存在该游戏的游戏卡 → 继续步骤 3
   - 如果游戏库中不存在该游戏 → 需要先创建游戏卡（使用 game-analyse 工作流）

### 步骤 3：创建/更新游戏卡
根据情况选择：

**情况 A：游戏卡已存在**
- 读取现有游戏卡内容
- 在"收录机制"部分添加新机制（格式：`- [品类]-游戏名-机制名`）

**情况 B：游戏卡不存在**
- 使用 game-analyse 工作流拆解该游戏
- 生成游戏卡后，继续步骤 4

### 步骤 4：创建机制卡
按上述模板格式化输出机制卡

### 步骤 5：写入文件
用户确认后写入 jizhi workspace：
- 机制：`/root/.openclaw/agents/jizhi/workspace/mechanisms/[游戏英文名]/[机制名].md`
- 游戏：`/root/.openclaw/agents/jizhi/workspace/games/[游戏英文名].md`

### 步骤 6：重新生成网页数据（增量更新）

**推荐：增量更新（只更新新增的文件）**：
```bash
cd /root/.openclaw/workspace/mechanism-lib-web
node update-data.js <新增机制卡路径> [新增游戏卡路径]
```

**备选：全量更新（重新解析所有文件）**：
```bash
cd /root/.openclaw/workspace/mechanism-lib-web
node generate-data.js
```

> 增量更新更快，适合单次入库；全量更新会重新校验所有关联，适合批量处理。

## 游戏卡检查清单

在创建机制卡前，必须完成以下检查：

- [ ] 游戏库中是否存在该游戏？
- [ ] 如果存在，获取游戏文件名（如 `300Heroes.md`）
- [ ] 如果不存在，使用 game-analyse 拆解游戏
- [ ] 机制卡的"来源游戏"字段是否与游戏卡一致？

## 入库筛选标准

### ✅ 值得入库的特征
1. **解决明确痛点** - 不是为了创新而创新
2. **非缝合设计** - 不是简单的组合
3. **可复制性** - 其他游戏可以借鉴

### ❌ 不入库的特征
1. 常见设计（抽卡、养成、战棋等）
2. 经典IP续作/资料片
3. 大厂热门但无创新

## 注意事项

- **一句话解释**：必填，用生活/商业类比讲清楚
- **演示素材**：机制卡必填，提供GIF或视频链接
- **核心标签**：必填。先查阅标准化标签表，选用已有标签归并，多个标签用空格分隔
- **来源游戏**：格式为 🔗 *游戏名*，与游戏库关联
- **收录机制**：格式为 [品类]-游戏名-机制名
