# 游戏机制条目生成器

将游戏拆解分析结果格式化为符合 jizhi 机制库标准的 Markdown 条目。

## 适用场景

- game-analyse-2-0 拆解完成后，需要生成入库格式
- 用户确认哪些支柱可以入库后，生成最终条目

---

## 标签标准化参考

**重要**：在创建机制卡时，必须使用标准化的标签体系。

请参考：[标签标准化参考.md](./标签标准化参考.md)

### 标签层级

1. **玩法层级标签**（必填）
   - 对抗类型、玩法人数规模、内容消耗模式、生命周期阶段
   - 操作属性、时间属性、玩法重度、玩法独立性

**⚠️ 人数规模自动提取**：从游戏卡的 X-Statement 中提取玩法规模，格式为：
   - `Total:X, Team:Y`（单局X人，单队Y人）
   - 例如："5v5 MOBA" → `Total:10, Team:5`
   - 例如："大逃杀100人" → `Total:100, Team:4`
   - 例如："4人合作PVE" → `Total:4, Team:4`
   - 如果跨度很大（如1-100人），写常见区间：`Total:1-10, Team:1`

2. **机制设计层级标签**（必填）
   - 原型品类、16乐趣标签、情绪反馈、交互关系
   - 玩家痛点标签、特色机制

## 机制库：设计详情标准卡 (Mechanism Card)

*核心资产库，采用高颗粒度拆解，确保不仅"能看懂"，还能"落地参考"。*

### ⚠️ 严格格式要求（必须遵守）

**标题格式**：
```
# [品类]-[游戏名]-[具体模块]
```
示例：`# [MOBA]-300英雄-7v7团战模式`

**禁止使用的格式**：
- ❌ `机制卡：皇室战争-圣水费用系统`
- ❌ `# 圣水费用系统`
- ❌ `## 机制名称：圣水费用系统`

**字段名称（必须使用标准名称）**：
| 标准字段 | 可接受变体 | 禁止使用 |
|----------|-------------|-----------|
| 案例名称 | - | 机制名称、标题 |
| 来源游戏 | 所属游戏 | - |
| 核心标签 | 标签 | - |
| 通俗转译 | - | 机制描述、机制拆解 |
| 一句话解释 | - | 通俗解释、简述 |

## 文件名命名规范（非常重要）

**机制卡文件名格式**：
```
[品类]-[游戏名英文/拼音]-[具体模块名].md
```

**命名规则**：

1. **前缀标签**（三选一）：
   - `[Excellent]` - 优秀机制，推荐入库
   - `[Qualify]` - 可选机制，需要评估
   - `[not_comfirmed]` - 未确认，需要进一步验证

2. **品类缩写**（使用标准英文缩写）：
   | 品类 | 缩写 |
   | :--- | :--- |
   | FPS | FPS |
   | MOBA | MOBA |
   | RPG | RPG |
   | Roguelike | Roguelike |
   | 大逃杀 | BR |
   | 搜打撤 | 撤离类 |
   | RTS | RTS |
   | 策略/战棋 | 策略 |
   | 模拟经营 | 模拟经营 |
   | 街机动作 | 街机 |
   | MMO | MMORPG |

3. **游戏名**：使用英文或拼音（与游戏卡文件名一致）
   - 例如：`DeltaForce`、`HonorOfKings`、`Palworld`

4. **模块名**：使用中文，简洁概括机制核心
   - 例如：`干员技能系统`、`缩圈博弈系统`、`武器多形态切换`

**正确示例**：
```
[Excellent][FPS]-DeltaForce-干员技能系统.md
[Qualify][BR]-PUBG-缩圈博弈系统.md
[Excellent][Roguelike]-Hades-武器多形态切换.md
[Qualify][策略]-AgeOfEmpires4-时代升级体系.md
```

**错误示例**：
```
❌ 干员技能系统.md          # 缺少前缀和品类
❌ FPS-干员技能.md          # 缺少游戏名
❌ 三角洲行动干员技能.md    # 缺少品类前缀
❌ [FPS]三角洲行动-干员技能.md  # 前缀格式错误
```

## 目录结构规范（非常重要）

**正确的目录结构**：
```
jizhi/workspace/
├── games/                      # 游戏卡目录（扁平结构）
│   ├── GenshinImpact.md       # 游戏卡
│   ├── DeltaForce.md
│   └── ...
└── mechanisms/                 # 机制卡目录（按游戏分文件夹）
    ├── GenshinImpact/          # 与游戏卡文件名对应
    │   ├── [Excellent]xxx.md
    │   └── [Qualify]xxx.md
    ├── DeltaForce/
    └── ...
```

**关键规则**：
- 游戏卡必须放在 `games/` 目录，文件名格式：`[游戏英文名].md`
- 机制卡必须放在 `mechanisms/[游戏英文名]/` 目录下
- 游戏英文名必须一致（如游戏卡是 `GenshinImpact.md`，机制卡目录必须是 `GenshinImpact/`）

## 使用流程

### 步骤 1：接收任务
接收 game-analyse-2-0 的分析结果，确认要入库的机制和游戏

### 步骤 2：检查目录和命名（新增）
**重要**：在入库前，必须验证目录结构和命名是否符合规范

1. **检查目标目录是否存在**：
   - 游戏卡应在：`/root/.openclaw/agents/jizhi/workspace/games/[游戏英文名].md`
   - 机制卡应在：`/root/.openclaw/agents/jizhi/workspace/mechanisms/[游戏英文名]/`

2. **如果目录/文件不存在**：
   - 游戏卡不存在 → 需要先创建游戏卡
   - 机制卡目录不存在 → 需要创建对应游戏文件夹

3. **命名规范检查**：
   - 游戏卡：`[游戏英文名].md`（如 `DeltaForce.md`）
   - 机制卡：`[前缀][品类]-[游戏英文名]-[模块名].md`（如 `[Excellent]FPS-DeltaForce-干员技能系统.md`）

### 步骤 3：检查现有机制库（重要）

**重要**：在创建机制卡前，必须先查询机制库是否已有相关机制

1. **查询方法**：
   - 查看 `/root/.openclaw/agents/jizhi/workspace/mechanisms/[游戏英文名]/` 目录
   - 或者搜索机制库网站：http://88.88.88.88:8888

2. **如果机制已存在**：
   - 更新现有机制卡（补充信息）
   - 在游戏卡的"收录机制"中关联

3. **如果机制不存在**：
   - 创建新的机制卡
   - 后续自动同步到游戏卡

### 步骤 4：检查游戏卡是否存在

1. **查找游戏库目录**：`/root/.openclaw/agents/jizhi/workspace/games/`
2. **检查方法**：
   - 如果游戏库中已存在该游戏的游戏卡 → 继续步骤 5
   - 如果游戏库中不存在该游戏 → 需要先创建游戏卡（使用 game-analyse 工作流）

### 步骤 5：创建/更新游戏卡
根据情况选择：

**情况 A：游戏卡已存在**
- 读取现有游戏卡内容
- 在"收录机制"部分添加新机制（格式：`- [前缀][品类]-游戏名-模块名`）
  - 例如：`- [Excellent]FPS-DeltaForce-干员技能系统`

**情况 B：游戏卡不存在**
- 使用 game-analyse 工作流拆解该游戏
- 生成游戏卡后，继续步骤 6

### 步骤 6：创建机制卡
按上述模板格式化输出机制卡

### 步骤 7：写入文件
用户确认后写入 jizhi workspace：

- **机制卡路径**：`/root/.openclaw/agents/jizhi/workspace/mechanisms/[游戏英文名]/[前缀][品类]-[游戏英文名]-[模块名].md`
  - 例如：`/root/.openclaw/agents/jizhi/workspace/mechanisms/DeltaForce/[Excellent]FPS-DeltaForce-干员技能系统.md`

- **游戏卡路径**：`/root/.openclaw/agents/jizhi/workspace/games/[游戏英文名].md`
  - 例如：`/root/.openclaw/agents/jizhi/workspace/games/DeltaForce.md`

**文件名生成公式**：
```
[前缀][品类]-[游戏英文名]-[模块名].md
```
- 前缀：`[Excellent]` / `[Qualify]` / `[not_comfirmed]`
- 品类：FPS / MOBA / RPG / Roguelike / BR / 撤离类 / RTS / 策略 / 模拟经营 / 街机 / MMORPG
- 游戏英文名：与游戏卡文件名一致（如 `DeltaForce`、`HonorOfKings`）
- 模块名：中文描述（如 `干员技能系统`、`缩圈博弈系统`）

### 步骤 8：重新生成网页数据（增量更新）

**推荐：增量更新（只更新新增的文件）**：
```bash
cd /root/.openclaw/workspace/mechanism-lib-web
node update-data.js <新增机制卡路径> [新增游戏卡路径]
```

**备选：全量更新（重新解析所有文件）**：
```bash
cd /root/.openclaw/workspace/mechanism-lib-web
node generate-data.js
```

> 增量更新更快，适合单次入库；全量更新会重新校验所有关联，适合批量处理。

## 游戏卡检查清单

在创建机制卡前，必须完成以下检查：

- [ ] 目录结构是否正确？（游戏卡在 `games/`，机制卡在 `mechanisms/[游戏名]/`）
- [ ] 游戏卡文件名是否与机制卡目录名一致？（如 `GenshinImpact.md` ↔ `GenshinImpact/`）
- [ ] 游戏库中是否存在该游戏？
- [ ] 如果存在，获取游戏文件名（如 `300Heroes.md`）
- [ ] 如果不存在，使用 game-analyse 拆解游戏
- [ ] 机制卡的"来源游戏"字段是否与游戏卡一致？

## 入库筛选标准

### ✅ 值得入库的特征
1. **解决明确痛点** - 不是为了创新而创新
2. **非缝合设计** - 不是简单的组合
3. **可复制性** - 其他游戏可以借鉴

### ❌ 不入库的特征
1. 常见设计（抽卡、养成、战棋等）
2. 经典IP续作/资料片
3. 大厂热门但无创新

## 注意事项

- **一句话解释**：必填，用生活/商业类比讲清楚
- **演示素材**：机制卡必填，提供GIF或视频链接
- **核心标签**：必填。先查阅标准化标签表，选用已有标签归并，多个标签用空格分隔
- **来源游戏**：格式为 🔗 *游戏名*，与游戏库关联
- **收录机制**：格式为 `[前缀][品类]-游戏名-模块名`
  - 例如：`[Excellent]FPS-DeltaForce-干员技能系统`
