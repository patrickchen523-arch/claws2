# game-entry-formatter 经验总结

## 技能简介
将游戏拆解分析结果格式化为 jizhi 机制库可入库的条目。支持机制卡（Mechanism Card）和游戏卡（Context Card）的标准化输出。

## 使用经验

### 1. 入库工作流

**完整流程**：
```
阶段一：收集（web_search + web_fetch）
阶段二：拆解（game-analyse-2-0）
阶段三：分析筛选（判断哪些机制值得入库）
阶段四：评分入库（game-entry-formatter + mechanism-scoring）
```

**关键规则**：
- 入库前先检查游戏卡是否存在
- 不存在则创建游戏卡
- 创建机制后同步更新游戏卡的"收录机制"
- 评分 < 3.5分 → 不通过入库，退回修改
- 评分 ≥ 3.5分 → 添加分级前缀并入库

### 2. 入库标准

**机制入库条件**：
1. 机制与主题高度统一
2. 有独特的策略维度
3. 机制足够抽象可复用
4. 有借鉴先例但有改进

**Excellent vs Qualify 判定**：
- **Excellent**：品类首创/标杆设计，有效解决核心痛点
- **Qualify**：有价值但非首创，可作为参考设计

### 3. 标准化标签

**玩法层级标签（8类）**：
1. 对抗类型（PVE/PVP/PvPvE）
2. 玩法人数规模（单局人数）
3. 内容消耗模式（可重复/UGC/一次性）
4. 生命周期阶段（持续运营/Season/关卡制）
5. 操作属性（即时操作/放置操作）
6. 时间属性（碎片化/中规中矩/重度）
7. 玩法重度（轻度/中度/重度）
8. 玩法独立性（高耦合/低耦合）

**标签格式规范**：
- 玩家痛点：`挫败感痛点`、`内容枯竭痛点`（必须以"痛点"结尾）
- 特色机制：`特色机制-失衡系统`（使用连字符）

### 4. 文件格式问题

**常见解析失败原因**：
1. 表格没有换行 - 使用正则 `\|([^\n|]+)\|([^\n|]*)\|` 拆分
2. 章节命名不一致：基本信息 vs 基础信息
3. 标签分隔符不统一：顿号、逗号、斜杠

**Fallback 策略**：
```javascript
// 从原始内容提取字段
const genreMatch = rawContent.match(/游戏类型\s*\|\s*([^|]+)/);
if (genreMatch) info.genre = genreMatch[1].trim();
```

### 5. 关联机制处理

**游戏卡关联机制**：
- 格式：`- 品类-游戏名-机制名`
- 验证：每次 generate-data.js 运行自动清理无效引用

**同步脚本**：
- 自动扫描机制库
- 向游戏卡添加新的关联
- 清理失效引用（严格匹配）

### 6. 封面图获取

**来源优先级**：
1. Steam header.jpg：`https://steamcdn-a.akamaihd.net/steam/apps/{appid}/header.jpg`
2. Google Play 图标
3. RAWG API

## 常见问题

### Q: 机制卡格式不统一
**A**: 使用标准模板，严格按照字段填写

### Q: 评分不通过
**A**: 检查是否符合入库标准，是否有创新点

### Q: 游戏卡不存在
**A**: 先创建游戏卡，再创建机制卡
