### 2.2 实体与组件

#### 实体定义

| 实体名称 | 类型 | 说明 |
|----------|------|------|
| | | |

#### 属性设计

```
EntityName {
    attribute_name: type    // 说明，取值范围 [min, max]
    attribute_name: type    // 说明
}
```

**示例**：
```
Pet {
    id: uint64              // 宠物唯一ID
    name: string            // 宠物昵称，最长16字符
    level: uint32           // 当前等级，范围 [1, 100]
    exp: uint64             // 当前经验值
    quality: enum           // 品质：白/绿/蓝/紫/橙
    skills: list<SkillID>   // 已学技能列表，最多6个
    hunger: float           // 饥饿度，范围 [0.0, 1.0]
}
```

## 审查检查清单

> 审查模式使用以下清单逐项核验策划案完整性。

### 结构完整性

- [ ] 概要设计：系统名称、目标、功能清单齐全
- [ ] 核心玩法：循环描述清晰，有实体/属性定义
- [ ] 状态机：所有状态、转换条件、边界条件列出
- [ ] 公式设计：每个公式有参数说明和取值范围
- [ ] UI 交互：界面布局、操作流程、跳转关系完整
- [ ] 美术音效：每个功能的资产需求已列出，使用规范 ID

### 逻辑一致性

- [ ] 属性取值范围无矛盾（如等级上限在不同章节一致）
- [ ] 状态机无孤岛状态（每个状态至少有一条进入和退出路径）
- [ ] 公式中的参数在实体属性中有定义
- [ ] 功能前置条件在其他功能中有产出/满足路径

### 数值完备性

- [ ] 所有属性有取值范围
- [ ] 资源产出与消耗有平衡考量
- [ ] 成长曲线有数据或公式支撑

### 边界条件

- [ ] 每个功能模块列出异常情况处理
- [ ] 考虑并发、网络、溢出等技术边界
- [ ] 考虑玩家极端操作（快速点击、断线重连等）
