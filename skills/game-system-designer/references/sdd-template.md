## 4 核心算法与伪代码

> 将 GDD 中的公式和规则转化为可实现的伪代码。
> 标注时间复杂度，便于程序评估性能。

### 4.1 核心逻辑

```pseudocode
function CoreFunction(params):
    // 前置检查
    if not validate(params):
        return ERROR_INVALID_PARAMS

    // 核心逻辑
    result = compute(params)

    // 持久化
    save_to_db(result)

    // 通知
    notify_client(result)

    return SUCCESS

// 时间复杂度: O(n)
// 空间复杂度: O(1)
```

```
    STATE_C = 2

```

### 5.2 状态转换表

| 当前状态 | 事件/条件 | 目标状态 | 守卫条件 | 副作用 |
|----------|-----------|----------|----------|--------|
| STATE_A | EVENT_X | STATE_B | level >= 10 | 发送通知 |
| STATE_B | EVENT_Y | STATE_A | 无 | 清理缓存 |

### 5.3 状态转换伪代码

```pseudocode
function transition(entity, event):
    current = entity.state

    switch (current, event):
        case (STATE_A, EVENT_X):
            if entity.level >= 10:
                entity.state = STATE_B
                notify(entity, STATE_CHANGED)
                return SUCCESS
            return ERROR_GUARD_FAILED

        case (STATE_B, EVENT_Y):
            entity.state = STATE_A
            clear_cache(entity)
            return SUCCESS

        default:
            return ERROR_INVALID_TRANSITION
```

```pseudocode
    if not validate(params):

    // Step 5: 持久化

    // Step 6: 通知与反馈
    notify_client(player, result)
    trigger_assets(result)  // VFX/SFX

    return SUCCESS

// 时间复杂度: O(?)
// 空间复杂度: O(?)
```

## 6 数据持久化

> 描述运行时数据如何落盘、缓存策略、数据一致性保障。

### 6.1 存储方案选型

| 数据类别 | 存储介质 | 读写模式 | 一致性要求 | 说明 |
|----------|----------|----------|------------|------|
| 玩家核心数据 | MySQL/MongoDB | 读多写少 | 强一致 | 角色属性、背包等 |
| 排行榜 | Redis Sorted Set | 读多写多 | 最终一致 | 定时持久化到 DB |
| 日志/埋点 | Kafka → 数据仓库 | 只写 | 无 | 异步写入 |
| 临时会话数据 | 内存 | 读写频繁 | 不持久化 | 进程重启丢失 |

### 6.2 缓存策略

| 缓存层 | 缓存对象 | 过期策略 | 更新方式 | 容量上限 |
|--------|----------|----------|----------|----------|
| 本地内存 (L1) | 配置表、热点数据 | 不过期/配置热更 | 主动刷新 | 按进程内存预算 |
| Redis (L2) | 玩家会话、排行榜 | TTL / LRU | 写穿透 (Write-Through) | 按集群容量 |

```
读请求路径:
  Client → L1 Cache hit? → 返回
                    miss → L2 Cache hit? → 回填 L1 → 返回
                                    miss → DB 查询 → 回填 L1+L2 → 返回

写请求路径:
  Client → 更新 DB → 更新 L2 → 失效 L1 → 返回
```

### 6.3 数据一致性保障

| 场景 | 一致性方案 | 说明 |
|------|------------|------|
| 单实体并发写 | 乐观锁 (version 字段) | 写入时校验版本号，冲突则重试 |
| 跨实体事务 | 本地事务 / Saga 模式 | 视业务复杂度选择 |
| 缓存与 DB 不一致 | 延迟双删 / 消息补偿 | 写 DB 后异步刷新缓存 |
| 宕机数据恢复 | WAL / Binlog 重放 | 恢复到最近一致点 |

### 6.4 数据生命周期

| 数据类型 | 保留策略 | 归档方式 | 清理触发 |
|----------|----------|----------|----------|
| 活跃玩家数据 | 永久 | - | - |
| 离线 >180天 | 归档 | 冷存储/压缩表 | 定时任务 |
| 操作日志 | 90天 | 按月归档到 OSS | 定时任务 |
| 临时/调试数据 | 7天 | 不归档 | TTL 自动过期 |
